# 1. Резюме и предпосылки

Проектируемая система обслуживает базу знаний из **бизнес-процессных сценариев** — документов вида «Создание платежа и холдирование (happy-path)»: с таблицами участников, пронумерованными шагами, переходами state machine, сквозными кодами-ссылками (`SQ-07`, `BR-PAY-07`) на другие документы, разделами явно исключённой функциональности и открытых вопросов.

Ключевые вводные:

- Нагрузка — до 100 запросов в день. Это не highload-задача: приоритет отдаётся точности, объяснимости и простоте эксплуатации, а не пропускной способности.
- LLM для генерации ответа уже выбрана и находится вне зоны проектирования этого документа.
- База знаний неоднородна: связная проза, таблицы, псевдо-диаграммы (текстовые), структурированные графы состояний, а в перспективе — логи, код, конфигурации.
- Домен чувствителен к точности: некорректная выдумка перехода state machine или фискального правила — это не стилистическая ошибка, а фактическая, с реальными последствиями (юридические/финансовые формулировки).

Главный архитектурный вывод: **чистый vector-only RAG здесь недостаточен**. Нужен гибридный retrieval, который сочетает семантический поиск по прозе с точным (exact-match/graph) поиском по структурированным сущностям — кодам, номерам шагов, состояниям — и генерацию, которая обязана различать три эпистемических статуса: *«описано»*, *«явно исключено»*, *«открытый вопрос»*.

# 3. Ingestion Pipeline (приём и нормализация)

## 3.1 Парсинг источников

| Тип источника | Инструмент | Особенности |
|---|---|---|
| PDF-сценарии | `pdfplumber` / `unstructured` | Извлечение текста с сохранением порядка блоков; таблицы — отдельным extractor'ом (`extract_tables()`), не просто as text |
| Markdown/DOCX | нативный парсер (`markdown-it`, `python-docx`) | Сохраняет заголовки/списки как структуру, не как plain text |
| JSON/БД | прямое чтение схемы | Не чанкуется как текст — хранится как структурированная запись с собственным индексом |
| Логи | построчный парсер + группировка по transaction/request id | Чанкуется по логической операции, не по количеству строк |
| Код | AST-парсер (`tree-sitter`) | Чанкуется по функции/классу, а не по произвольному количеству строк |

Для сценариев вида «happy-path» пайплайн должен уметь выделить пять типов блоков одного документа:

1. **Заголовок и метаданные** (имя, дата версии, ссылка на общий индекс конвенций).
2. **Область покрытия** («Покрывает: …») — отдельное поле, а не часть общего текста.
3. **Таблица участников** — парсится как таблица (роль, система, обязанности), сохраняется и как структурированная запись, и как денормализованный текст для эмбеддинга.
4. **Фазы + пронумерованные шаги**, привязанные к диаграмме — каждый шаг получает `phase_id`, `step_number`, `actor`.
5. **State machine переходы** — извлекаются регулярным парсером (`X → Y` паттерны) в отдельную таблицу `transitions(from_state, to_state, trigger, scenario_id)`.
6. **Negative scope** («Что НЕ требуется») и **Open edge cases** («Граничные ситуации, не покрытые») — размечаются отдельным `section_type`, чтобы генерация могла явно на них ссылаться.
7. **Сквозные коды** (`SQ-*`, `BR-*`) — извлекаются regex'ом (`\b(SQ|BR)-[A-Z0-9-]+\b`) и записываются как рёбра графа `references(source_doc, code, context_snippet)`.

## 3.2 Нормализация

- Единый внутренний формат документа: `{doc_id, version, title, coverage, sections[], transitions[], participants[], references[]}`.
- Версионность: имя файла содержит дату (`2026-05-27`) — при обновлении сценария старая версия не удаляется, а помечается `superseded_by`, чтобы retrieval не путал устаревшие и актуальные шаги.
- Нормализация терминов (например, «холдирование» / «hold» / «холд») через словарь синонимов домена — важно для sparse-поиска.

## 3.3 Чанкование

Гранулярность — **на уровне шага/раздела, а не фиксированного окна токенов**:

- Один чанк = один шаг сценария (с включённым контекстом фазы и предыдущего шага для связности) либо один смысловой раздел (`coverage`, `negative_scope`, `open_questions`).
- Каждый чанк дублирует в метаданных: `scenario_id`, `phase`, `step_number`, `section_type`, `related_codes[]`, `version_date`.
- Таблицы (участники, state machine) не чанкуются построчно — хранятся целиком как структурированная сущность плюс denormalized-текстовое summary для попадания в векторный поиск.

# 4. Хранилище знаний

Три параллельных хранилища с общим `doc_id`/`scenario_id` как ключом связи:

## 4.1 Векторное хранилище (semantic layer)

- **Выбор БД**: **Qdrant** (self-hosted, Docker) или managed **pgvector** поверх уже используемой PostgreSQL — при нагрузке 100 запросов/день и, вероятно, сотнях-низких тысячах документов, нет причины брать Pinecone/Weaviate/Milvus с их операционной сложностью и стоимостью. `pgvector` предпочтителен, если в организации уже есть PostgreSQL — это минимизирует число систем в эксплуатации; Qdrant предпочтителен, если нужен более гибкий hybrid search "из коробки" (payload-фильтры + BM25).
- **Embeddings-модель**: мультиязычная модель (домен — русскоязычные документы с англоязычными терминами: `payment_intent`, `webhook`, `capture`) — например, `bge-m3` (поддерживает dense + sparse + ColBERT-style multi-vector в одной модели) или `text-embedding-3-large`, если уже используется OpenAI-совместимый провайдер. `bge-m3` предпочтителен, так как совмещает dense/sparse и снижает число компонентов в стеке.
- Хранится: текст чанка, embedding, метаданные из 3.3.

## 4.2 Структурное хранилище (structured layer)

- Обычная реляционная БД (PostgreSQL) с таблицами:
  - `scenarios(id, title, coverage, version_date, superseded_by)`
  - `participants(scenario_id, role, system, responsibility)`
  - `transitions(scenario_id, from_state, to_state, trigger, phase)`
  - `sections(scenario_id, section_type, text, related_codes[])`
- Позволяет делать точные SQL-запросы («какие переходы ведут в `Захолдирован`») без обращения к LLM вообще — для части вопросов это и есть весь ответ.

## 4.3 Граф ссылок (reference graph)

- Легковесный граф (можно на той же PostgreSQL через таблицу `references(from_doc, to_code, snippet)`, без отдельной graph DB — при таком объёме Neo4j избыточен).
- Позволяет резолвить `SQ-07`, `BR-PAY-07` в исходный документ/раздел, даже если запрос пользователя написан в терминах кода, а не текста.

# 5. Retrieval и ранжирование

## 5.1 Гибридная стратегия

Запрос пользователя параллельно обрабатывается тремя путями:

1. **Точный путь**: regex-детекция кодов (`SQ-\d+`, `BR-PAY-\d+`) или номеров шагов в запросе → прямой lookup в structured layer / reference graph. Если найдено — это высокоприоритетный источник контекста.
2. **Sparse-путь** (BM25/keyword): для доменных терминов, точных названий систем (`CloudPayments`, `Cloud Kassir`), точных значений статусов (`Захолдирован`).
3. **Dense-путь** (vector similarity): для семантически похожих, но иначе сформулированных вопросов.

## 5.2 Ранжирование

- Reciprocal Rank Fusion (RRF) объединяет результаты трёх путей — не полагаемся только на cosine similarity.
- Точные совпадения по кодам/номерам шагов получают фиксированный boost (например, всегда попадают в топ контекста, если релевантны запросу) — потому что для этого домена точный код важнее семантической близости.
- Reranker (cross-encoder, например `bge-reranker-v2-m3`) применяется к top-20 кандидатам перед финальной сборкой контекста — оправдан при таком низком QPS, так как задержка не критична, а точность важна.

## 5.3 Обработка смешанных типов данных при поиске

- Текстовые чанки → dense + sparse.
- Таблицы (участники, state machine) → structured query по совпадению сущности (роль, статус) плюс denormalized текстовое summary в векторном индексе как "мостик" для семантического поиска.
- Логи/код (при расширении) → отдельный индекс с иной схемой метаданных (timestamp/transaction_id для логов, symbol/file_path для кода) — не смешивать в одном векторном пространстве с прозой сценариев без явного разделения через `doc_type` фильтр.

# 6. Генерация ответа

## 6.1 Структура промпта для LLM

Контекст, передаваемый в LLM, размечен по трём эпистемическим категориям, и промпт explicit требует их не смешивать:

- **"Описано в сценарии"** — цитируется с указанием `scenario_id` + `step_number`/`section`.
- **"Явно исключено"** (из раздела negative scope) — LLM обязана ответить в духе «это вне рамок данного сценария», а не додумывать поведение.
- **"Открытый вопрос / не покрыто"** — LLM обязана явно сказать, что это описано как нерешённый edge case (со ссылкой на код открытого вопроса, если есть), а не предлагать собственную интерпретацию как факт.

## 6.2 Контроль галлюцинаций

- **Запрет на "достраивание" state machine**: промпт явно указывает, что переходы состояний нужно брать только из structured layer (`transitions`), а не выводить логически из прозы.
- **Обязательное цитирование**: каждое фактическое утверждение в ответе должно сопровождаться ссылкой на `scenario_id`+`step`/`section`; ответы без возможности сослаться на источник помечаются как low-confidence.
- **Self-consistency check** (опционально при таком низком QPS это недорого): для критичных вопросов (фискализация, переходы состояний) можно сгенерировать ответ дважды с чуть разным контекстным окном и сравнить на предмет расхождений перед выдачей.

# 7. Валидация и обратная связь

При объёме 100 запросов/день полноценный автоматизированный ML-цикл обратной связи избыточен — целесообразнее:

- **Ручная/полуручная разметка**: простой UI (или таблица) для оценки ответа (correct / partially correct / hallucinated / missing context) — это management-нагрузка, посильная при таком объёме.
- **Citation-check** как автоматическая метрика: доля ответов, где все фактические утверждения подкреплены валидной ссылкой на существующий `scenario_id`/`step`.
- **Regression-набор**: по мере роста базы — фиксированный набор вопросов с эталонными ответами (особенно про state transitions и negative scope), прогоняемый при каждом обновлении retrieval-логики или embeddings-модели.
- **Метрики**:
  - Recall@k для точных кодов (должен быть близок к 100%, это не "мягкая" семантика).
  - Доля ответов с корректным разграничением "описано/исключено/открытый вопрос".
  - Доля галлюцинированных переходов state machine (целевое значение — 0).

# 8. Масштабируемость и узкие места

## 8.1 Что проектируем "на вырост" уже сейчас

- Схема метаданных чанков и граф ссылок — их дорого будет переделывать при росте числа сценариев, поэтому фиксируем сразу.
- Версионность документов — при росте частоты обновлений сценариев это станет критичным раньше, чем вырастет QPS.

## 8.2 Что сознательно упрощаем при текущей нагрузке

- Единый инстанс `pgvector`/Qdrant без шардирования.
- Синхронная обработка запроса (без очередей) — оправдано при 100 запросов/день.
- Ручная валидация вместо автоматизированного feedback loop.

## 8.3 Вероятные узкие места при росте нагрузки

| Узкое место | Когда проявится | Решение |
|---|---|---|
| Синхронный reranker на каждый запрос | Рост QPS в десятки раз | Кэширование частых запросов, асинхронная обработка с очередью |
| Ручная разметка обратной связи | Рост объёма запросов | Автоматизация части проверок (citation-check уже автоматизирован) |
| Единый PostgreSQL для vector + structured + graph | Рост объёма документов на порядки | Вынос vector layer в специализированную БД (Qdrant/Milvus) |
| Regex-based извлечение кодов при усложнении формата документов | Изменение шаблона документов | Переход на LLM-assisted extraction с валидацией по схеме |

# 9. Итоговый технологический стек

| Компонент | Выбор | Обоснование |
|---|---|---|
| Векторная БД | Qdrant (или pgvector при наличии PostgreSQL) | Простота эксплуатации при низкой нагрузке, hybrid search из коробки |
| Embeddings | `bge-m3` | Мультиязычность + dense/sparse в одной модели |
| Reranker | `bge-reranker-v2-m3` | Приемлемая задержка при низком QPS, ощутимый прирост точности |
| Структурное хранилище | PostgreSQL | Точные запросы по state machine/участникам без обращения к LLM |
| Граф ссылок | Таблица в PostgreSQL | Neo4j избыточен при таком объёме |
| Оркестрация retrieval | Кастомный слой (LangChain/LlamaIndex как база, с кастомной hybrid-логикой) | Готовые библиотеки не покрывают доменный three-way retrieval "из коробки" |
| Извлечение сущностей (коды, шаги, state machine) | Regex + правило-ориентированный парсер, с LLM-fallback на сложных случаях | Дёшево, предсказуемо, легко тестировать |
